## Пайплайн (один цикл)

### 1. Fetcher
- Подключается к каждому чату через userbot
- Читает `data/checkpoints.json` — ID последнего прочитанного сообщения
- Запрашивает новые сообщения после чекпоинта (ADR-001)
- Пустые сообщения (без текста) пропускает
- Возвращает: `[{id, chat_id, chat_title, sender, sender_name, text, date, link}]`
- Сохраняет новые чекпоинты

### 2. Keyword Scanner
- Нечёткий поиск (rapidfuzz, threshold 75%)
- 9 категорий, ~380 ключевых слов
- Автоматическая транслитерация ru↔en
- Отфильтровывает ~90% мусора

### 3. AI Analyzer
- LLM получает **все сообщения** батча (ADR-002), а не только keyword matches
- Чанки: max 30 сообщений / 15 000 символов (ADR-003)
- Модель: `glm-4.5-air` через z.ai API
- Температура: 0.1
- Возвращает: `[{id, is_lead, reason}]` — только лиды
- Fallback: при невалидном JSON — повторный запрос с упрощённым промптом

### 4. Merge + Dedup
- Keyword и AI результаты объединяются по message_id
- Если оба нашли — AI-причина обогащает keyword-результат
- Дедупликация по `data/leads.json` (уже отправленные не повторяются)

### 5. Notifier
- Форматирует сообщение с: чат, автор, текст, категория, AI-причина, ссылка
- Батчит по `MAX_LEADS_PER_MESSAGE` (10 лидов на сообщение)
- Отправляет через бота (`@ext_team_f_bot`) в управляющую группу

### 6. Obsidian
- Сохраняет отчёт: `05-Journal/oleg-telegram-leads/YYYY-MM-DD.md`
- Append к файлу дня (если уже существует)

## Управляющая группа

Бот `@ext_team_f_bot` обрабатывает через event handler:

| Команда | Описание |
|---------|----------|
| `/status` | Статус: кол-во чатов, лидов, модель |
| `/run` | Запустить цикл вручную |
| `/list` | Список мониторимых чатов |
| `/add <chat>` | Добавить чат (по ID или username) |
| `/remove <chat_id>` | Удалить чат |
| `/leads` | Последние 10 найденных лидов |
| `/help` | Справка |

**Обработка пересылок:** Олег пересылает сообщение из чата → бот принимает, сохраняет для будущего размечивания (категория: лид / не лид).

## Ограничения Bot API

- Бот **не может** читать историю сообщений (`iter_messages` / GetHistoryRequest)
- Бот **может** отправлять сообщения (`send_message`)
- Бот **может** принимать новые сообщения через event handler (`events.NewMessage`)
- Для чтения мониторимых чатов используется userbot

## Масштабирование

Текущая архитектура (poll-based daemon) реалистична для:

| Метрика | Лимит |
|---------|-------|
| Чатов | 10-50 |
| Сообщений/день | до 50 000 |
| AI запросов/цикл | 3-10 |

Для большего масштаба нужна queue-based архитектура (ADR-012).

## Обработка ошибок

- **FloodWait** — ожидание до 300 сек, затем повтор
- **AI timeout** — httpx timeout 60 сек, при ошибке — лог и пропуск батча
- **Невалидный JSON от AI** — fallback-промпт, повтор
- **Падение userbot** — reconnect в следующем цикле
- **Graceful shutdown** — SIGINT/SIGTERM → завершение текущего цикла и выход
