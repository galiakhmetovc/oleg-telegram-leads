## Пайплайн (один цикл)

### 1. Fetcher
- Подключается к каждому чату через userbot
- Читает `data/checkpoints.json` — ID последнего прочитанного сообщения
- **Direct fetch** для `pending_scan_ids` и `initial_scan_msg_id` (целевые сообщения через `get_messages(ids=...)`)
- **Основной запрос** — `iter_messages(min_id=checkpoint)` для новых сообщений после чекпоинта
- Читает `msg.text or msg.message or msg.caption` (поддержка media+caption)
- Пропускает сообщения без текста (логирует как DEBUG)
- Возвращает: `[{id, chat_id, chat_title, sender, sender_name, text, date, link}]`
- Сохраняет новые чекпоинты (max ID из полученных сообщений)

### 2. Keyword Scanner
- Точный поиск по отдельным словам (word boundaries) — никаких частичных совпадений внутри слов
- Нечёткий поиск (rapidfuzz fuzz.WRatio, threshold 84%) для многословных фраз
- 9 категорий, ~380 ключевых слов из `config.py` + `keywords.md`
- Автоматическая транслитерация ru↔en
- **Только логирование** — keyword-совпадения НЕ отправляются как лиды (ADR-016)

### 3. AI Analyzer
- LLM получает **все сообщения** батча (ADR-002), а не только keyword matches
- Чанки: max 15 сообщений / 5 000 символов (ADR-003)
- Модель: `glm-4.5` через z.ai API
- Температура: 0.1
- **Thinking mode отключён** (`"thinking": {"type": "disabled"}`) — не нужен для классификации
- Few-shot промпт: 7 примеров лидов + 3 контрпримера
- 2 попытки получения JSON, fallback-промпт при неудаче
- Возвращает: `[{id, reason}]` — только AI-подтверждённые лиды

### 4. Merge + Dedup
- AI-результаты — основной источник лидов
- Keyword-совпадения обогащают AI-лиды (показывают `🔑 Слова:` в уведомлении)
- Дедупликация по `data/leads.json` (уже отправленные не повторяются)
- При `/recheck` — дедуп bypass (перепроверка конкретных сообщений)

### 5. Notifier
- Отправляет **только AI-подтверждённые лиды** (keyword-only не отправляются)
- Форматирует сообщение с: чат, автор, текст, keyword-слова, AI-причина, ссылка
- Батчит по `MAX_LEADS_PER_MESSAGE` (10 лидов на сообщение)
- Отправляет через бота (`@ext_team_f_bot`) в управляющую группу
- @mention Антона при найденных лидах

### 6. Obsidian (отключён — sync/async conflict)
- `_save_to_obsidian()` существует в pipeline.py
- Вызывается как синхронная функция (не `await`)
- Может быть повторно включена после фикса

## Управляющая группа

Бот `@ext_team_f_bot` обрабатывает через event handler:

| Команда | Описание |
|---------|----------|
| `/status` | Статус: кол-во чатов, лидов, модель |
| `/run` | Запустить цикл вручную |
| `/list` | Список мониторимых чатов |
| `/add <chat>` | Добавить чат (по ID или username) |
| `/remove <chat_id>` | Удалить чат |
| `/reset <chat_id>` | Сбросить чекпоинт чата |
| `/recheck <id...>` | Перепроверить конкретные сообщения через AI |
| `/leads` | Последние 10 найденных лидов |
| `/help` | Справка |

**Обработка ссылок:** `t.me/chat/123` — добавить чат, checkpoint = msg_id - 1, initial_scan_msg_id = msg_id

**Обработка пересылок:** пересылка из чата → добавить чат / сохранить пример лида

## Ограничения Bot API

- Бот **не может** читать историю сообщений (`iter_messages` / GetHistoryRequest)
- Бот **может** отправлять сообщения (`send_message`)
- Бот **может** принимать новые сообщения через event handler (`events.NewMessage`)
- Для чтения мониторимых чатов используется userbot

## Масштабирование

Текущая архитектура (poll-based daemon) реалистична для:

| Метрика | Лимит |
|---------|-------|
| Чатов | 10-50 |
| Сообщений/день | до 50 000 |
| AI запросов/цикл | 3-10 |

Для большего масштаба нужна queue-based архитектура (ADR-012).

## Обработка ошибок

- **FloodWait** — ожидание до 300 сек, затем повтор
- **AI timeout** — httpx timeout 60 сек, при ошибке — лог и пропуск батча
- **Невалидный JSON от AI** — fallback-промпт, 2 попытки
- **Падение userbot** — reconnect в следующем цикле
- **Graceful shutdown** — SIGINT/SIGTERM → завершение текущего цикла и выход
- **checkpoints.json не dict** — автосброс в `{}` с предупреждением (ADR-017)
