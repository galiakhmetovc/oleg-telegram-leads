# Архитектурные решения — Telegram Leads Finder

## ADR-001: Userbot (Telethon) вместо Bot API

**Дата:** 2025-07-17
**Статус:** принято
**Контекст:** Нужен доступ к открытым Telegram-группам. Bot API требует добавления бота в каждый чат, чтение истории ограничено.
**Решение:** Используем Telethon (userbot) для чтения сообщений. Аккаунт `@krab_ai_agent`.
**Ограничение:** Userbot может быть заблокирован Telegram при слишком частых запросах. FloodWait обрабатывается автоматически.

---

## ADR-002: Keywords + LLM — двухэтапная фильтрация

**Дата:** 2025-07-17
**Статус:** принято → **обновлено** (ADR-016)
**Контекст:** Отправка всех сообщений в LLM дорого и медленно. Только keywords — много пропусков (контекст без явных ключевых слов).
**Решение:** Двухэтапная схема:
1. Keyword scanner (rapidfuzz, ~380 слов) — быстрая фильтрация
2. LLM получает **все сообщения** батча — семантический анализ
3. **Только AI-подтверждённые результаты** отправляются как лиды (keyword-only не отправляются)

## ADR-003: Чанки по количеству и символам

**Дата:** 2025-07-17
**Статус:** принято → **обновлено**
**Контекст:** LLM имеет лимит контекста. Нельзя отправить 10 000 сообщений разом.
**Решение:** Двойное ограничение:
- `BATCH_SIZE` = 15 сообщений в чанк
- `BATCH_CHAR_LIMIT` = 5 000 символов в чанк
- При превышении лимита символов чанк разбивается дальше

## ADR-004: z.ai (OpenAI-совместимый API)

**Дата:** 2025-07-17
**Статус:** принято → **обновлено** (ADR-007, ADR-018)
**Контекст:** Нужна LLM для семантического анализа на русском.
**Решение:** z.ai API (OpenAI-совместимый). URL: `https://api.z.ai/api/coding/paas/v4` (не менять!).
- Модель: `glm-4.5` (тестировалась на glm-5.1, glm-4.5-air, glm-4.5 — все работают)
- Температура: 0.1
- Thinking mode: отключён (`"thinking": {"type": "disabled"}`)

## ADR-005: Первый запуск — не читать историю

**Дата:** 2025-07-17
**Статус:** принято
**Контекст:** При первом запуске нельзя читать всю историю чатов — будет лавина ложных срабатываний.
**Решение:** При отсутствии чекпоинта для чата — пропускаем чат (пустой checkpoint = 0).

## ADR-009: Daemon-режим (единый процесс)

**Дата:** 2026-04-27
**Статус:** принято
**Контекст:** Pipeline должен работать постоянно.
**Решение:** Единый daemon-процесс с двумя параллельными задачами:
1. **Bot event handler** — слушает управляющую группу (команды, пересылки, ссылки)

## ADR-016: Только AI-подтверждённые лиды

**Дата:** 2026-06-28
**Статус:** принято
**Контекст:** Keyword scanner давал ложные срабатывания («фотографы», «фигурные наличники дверей»). Пользователю нужен фильтрованный результат, а не все keyword matches.
**Решение:**
- Уведомления отправляются **только для AI-подтверждённых лидов**
- Keyword-совпадения только логируются и обогащают AI-лиды (показывают `🔑 Слова:`)
- Keyword fallback из enriched_leads убран
- Статус-репорт упрощён: `✉️ X сообщений | 🤖 Y AI | 🎯 Z лидов | 📂 W чатов`

## ADR-017: Защита от невалидных checkpoints.json

**Дата:** 2026-06-28
**Статус:** принято
**Контекст:** При сбросе данных бот записывал `[]` (list) вместо `{}` (dict) в checkpoints.json → `TypeError: list indices must be integers or slices, not str`.
**Решение:** `load_checkpoints()` проверяет `isinstance(data, dict)`, при несовпадении — сбрасывает в `{}` с предупреждением в лог.

## ADR-018: Thinking mode отключён

**Дата:** 2026-06-28
**Статус:** принято
**Контекст:** GLM модели по умолчанию включают thinking mode — генерируют рассуждения перед ответом. Для простой JSON-классификации это лишние токены и задержка.
**Решение:** В payload API добавлено `"thinking": {"type": "disabled"}`. Документация GLM-4.7 подтверждает поддержку этого параметра.

## ADR-020: Direct fetch для целевых сообщений

**Дата:** 2026-06-28
**Статус:** принято
**Контекст:** Telegram `iter_messages(min_id=X)` пропускает сообщения, если между checkpoint и текущим были удалённые сообщения. В чатах с активной модерацией это частая проблема.
**Решение:** При добавлении чата через ссылку:
- `initial_scan_msg_id` сохраняется в `chats.json`
- Fetcher сначала делает `get_messages(ids=target_id)` — прямое чтение
- Затем `iter_messages(min_id=checkpoint)` для остальных новых сообщений
- При `/recheck` — аналогично через `pending_scan_ids`

---

## ADR-021: Caption extraction для media-сообщений

**Дата:** 2026-06-28
**Статус:** принято
**Контекст:** Сообщения с фото+caption имели `msg.text=None` — fetcher их пропускал.
**Решение:** `msg_text = msg.text or getattr(msg, "message", None) or getattr(msg, "caption", None) or ""`

## ADR-022: FETCH_LIMIT увеличен до 500

**Дата:** 2026-06-28
**Статус:** принято
**Контекст:** При добавлении чата через старую ссылку (msg_id=716254, текущий msg_id=718380) fetcher с `FETCH_LIMIT=100` мог не получить все сообщения.
**Решение:** `FETCH_LIMIT = 500` (было 100).
