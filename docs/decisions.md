# Архитектурные решения — Telegram Leads Finder

## ADR-001: Userbot (Telethon) вместо Bot API

**Дата:** 2025-07-17
**Статус:** принято
**Контекст:** Нужен доступ к открытым Telegram-группам. Bot API требует добавления бота в каждый чат, чтение истории ограничено.
**Решение:** Используем Telethon (userbot) для чтения сообщений. Аккаунт `@krab_ai_agent`.
**Ограничение:** Userbot может быть заблокирован Telegram при слишком частых запросах. FloodWait обрабатывается автоматически.

---

## ADR-002: Keywords + LLM — двухэтапная фильтрация

**Дата:** 2025-07-17
**Статус:** принято
**Контекст:** Отправка всех сообщений в LLM дорого и медленно. Только keywords — много пропусков (контекст без явных ключевых слов).
**Решение:** Двухэтапная схема:
1. Keyword scanner (rapidfuzz, ~380 слов) — быстрая фильтрация ~90% мусора
2. LLM получает **все сообщения** батча (не только keyword matches) — для контекста

---

## ADR-003: Чанки по количеству и символам

**Дата:** 2025-07-17
**Статус:** принято
**Контекст:** LLM имеет лимит контекста. Нельзя отправить 10 000 сообщений разом.
**Решение:** Двойное ограничение:
- `BATCH_SIZE` = 30 сообщений в чанк
- `BATCH_CHAR_LIMIT` = 15 000 символов в чанк
- При превышении лимита символов чанк разбивается дальше

## ADR-004: z.ai (OpenAI-совместимый API)

**Дата:** 2025-07-17
**Статус:** принято
**Контекст:** Нужна LLM для семантического анализа на русском.
**Решение:** z.ai API (OpenAI-совместимый), модель `glm-4.5-air`. Температура 0.1. Легко заменить на другую модель.

## ADR-005: Первый запуск — не читать историю

**Дата:** 2025-07-17
**Статус:** принято
**Контекст:** При первом запуске нельзя читать всю историю чатов — будет лавина ложных срабатываний.
**Решение:** При отсутствии чекпоинта для чата — запоминаем ID последнего сообщения как стартовую точку. История не читается.

**Статус:** принято

## ADR-009: Daemon-режим (единый процесс)

**Дата:** 2026-04-27
**Статус:** принято
**Контекст:** Pipeline должен работать постоянно: слушать управляющую группу и запускать циклы по расписанию.
**Решение:** Единый daemon-процесс с двумя параллельными задачами:
1. **Bot event handler** — слушает управляющую группу (команды, пересылки)
2. **Scheduler loop** — каждые `POLL_INTERVAL` секунд запускает полный цикл (fetch → scan → analyze → notify)

Два Telegram-клиента в одном процессе:
- **Userbot** (`@krab_ai_agent`) — ТОЛЬКО чтение чатов
- **Bot** (`@ext_team_f_bot`) — управляющая группа (event handler для команд + send_message для уведомлений)

Заменяет ADR-008 (отдельный scheduler teamD). Daemon сам управляет расписанием.

## ADR-010: Отдельный бот для управляющей группы

**Дата:** 2026-04-27
**Статус:** принято
**Контекст:** Userbot не может использовать bot events. Для управления группой нужен бот с event handler.
**Решение:**
- Бот `@ext_team_f_bot` (Bot API) — обслуживает управляющую группу
- Bot event handler слушает команды (/status, /run, /add, /remove и т.д.)
- Bot send_message отправляет уведомления о лидах
- **Важно:** Bot API не поддерживает `iter_messages` (GetHistoryRequest) — бот не может читать историю. Для чтения команд используется bot event handler (NewMessage event).
- Userbot `@krab_ai_agent` — ТОЛЬКО чтение мониторимых чатов, не участвует в группе

## ADR-011: Результаты в Obsidian

**Дата:** 2026-04-27
**Статус:** принято
**Контекст:** Нужна история результатов для анализа и аудита.
**Решение:** После каждого цикла сохраняется отчёт в Obsidian:
- Путь: `05-Journal/oleg-telegram-leads/YYYY-MM-DD.md`
- Содержание: дата, время, количество сообщений, keyword matches, найденные лиды
- Если файл за день уже существует — добавляется новый блок (append)

## ADR-012: Ограничение текущей архитектуры

**Дата:** 2026-04-27
**Статус:** принято
**Контекст:** Нужно понимать, при каких нагрузках текущая архитектура перестаёт работать.
**Решение:**

| Параметр | Текущее значение | Предел |
|----------|-----------------|--------|
| Чатов | 10-50 | ~50 (далее — flood wait) |
| Сообщений/день | до 50 000 | ~100 000 |
| Keyword scan | O(n×m) | OK для 50 чатов |
| AI запросов/цикл | ~3-10 | зависит от z.ai rate limit |

**Для 1000 чатов × 10 000 сообщений/день (10 млн) нужна queue-based архитектура:**
- Telethon events вместо poll (реальное время)
- Redis/RabbitMQ queue
- Worker pool для keyword scan
- Rate limiter для z.ai API
- Multi-session Telethon (разные аккаунты)

Текущая архитектура — MVP для 10-50 чатов. Масштабирование — отдельная задача.

## ADR-013: Добавление чатов через пересылки

**Дата:** 2026-04-27
**Статус:** принято
**Контекст:** Нужен удобный способ добавления чатов в мониторинг с телефона. Ручное редактирование config неудобно.
**Решение:** Два сценария обработки пересылок в управляющей группе:

| Чат в мониторинге? | Действие |
|---|---|
| ❌ Нет | **Сценарий 1** — добавить чат, чекпоинт = ID пересланного сообщения |
| ✅ Да | **Сценарий 2** — сохранить как пример лида для обучения AI |

Критерий различия: чат уже есть в `data/chats.json` или нет.
Примеры лидов сохраняются в `data/lead_examples.json`.

## ADR-014: @mention Антона при найденных лидах

**Дата:** 2026-04-27
**Статус:** принято
**Контекст:** Антон должен быть уведомлён о найденных лидах явно (не просто сообщение в группе).
**Решение:** При отправке уведомлений о лидах в первое сообщение добавляется `tg://user?id={ANTON_TELEGRAM_ID}` — это вызывает пуш-уведомление и подсвечивает Антона в группе.
Если лидов нет — отправляется обычный статусный репорт без mention.

---

## ADR-015: Статус-репорт каждый цикл

**Дата:** 2026-04-27
**Статус:** принято
**Контекст:** Нужно понимать, что система работает, даже если лидов не найдено.
**Решение:** После каждого цикла в управляющую группу отправляется краткий статус:
- Сколько чатов проверено
- Сколько сообщений загружено
- Keyword matches / AI matches
- Количество найденных лидов
- Если лиды найдены — отдельное сообщение с @mention Антона
